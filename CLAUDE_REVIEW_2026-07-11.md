# Обзор кода UiController — найденные проблемы и правки

**Дата:** 2026-07-11
**Объём:** HEAD `230785a` (до него `6f6eeef` «Phase 3+4 Full feature completion»), ~92 `.cs`, ~15.5к строк.
**Метод:** два независимых прохода по коду, находки первого прохода перепроверены вторым по `файл:строка`.
**Формат:** внешнее ревью — только по факту кода, конструктивно. Ниже — что работает не так, как заявлено в README, и как поправить. Код в репозитории не менялся.

---

## Сильные стороны (чтобы не сломать при рефакторинге)

- JSON-контракт ответа (`CommandResult` + `UiStateDiff` + self-describing ошибки) — аккуратно и удобно для программного потребителя.
- Каскад деградации поиска элемента (UIA AutomationId/Name → locale → ai-find → UiMap → координаты → Win32 → WinAppDriver).
- `UiMap` (YAML Page Object с версионными оверрайдами) + `LocaleMap` (RU/EN).
- `cv-capture` — скриншот региона/элемента в файл + `<name>.meta.json`.
- Единый лог ошибок UIA (`SafeExtensions`) вместо молчаливых `catch{}`.

Правки ниже это ядро не затрагивают — они закрывают разрыв между README и фактическим поведением.

## Проблемы (по серьёзности, файл:строка)

### 1. `--uia-only` / headless-скриншоты не работают вообще
`ScreenshotHelper.GetWadScreenshot()` (`ScreenshotHelper.cs:147-152`) читает `Program.WadClient`, но это поле **нигде не присваивается** — единственные создания клиента локальные (`using var client = new WinAppDriverClient()` в `WinAppDriverCommands.cs:22,56,109`, уничтожаются на выходе из команды). Итог: при `--uia-only` любой скриншот молча возвращает `null`, даже при живом WinAppDriver. Заявленная в справке (`Program.cs:365`) поддержка headless — не включается.
**Фикс:** присваивать `Program.WadClient` при `--uia-only` (или сделать поле-фабрику с ленивой инициализацией). Дополнительно стоит добавить канал `PrintWindow(PW_RENDERFULLCONTENT)` — он снимает конкретный HWND даже перекрытый/фоновый, без зависимости от WinAppDriver.

### 2. Таймауты UIA фиктивные — нет защиты от зависшего дерева
`AutomationHelper.SafeGetChildren` (`AutomationHelper.cs:10-25`) создаёт `CancellationTokenSource(timeoutMs)`, но токен никуда не передаётся — блокирующий COM-вызов `FindAllChildren()` может висеть бесконечно. Все «таймауты» 3000/5000/8000 мс по AutomationHelper декоративные; watchdog-а от зависшего UIA-дерева нет.
**Фикс:** выполнять `FindAllChildren()` в задаче с реальным дедлайном (`Task.Run` + `Wait(timeout)` или отдельный поток с прерыванием) и общий бюджет времени на команду.

### 3. Параллельный обход живого COM-дерева
`AutomationHelper.FindControlsByName` (`AutomationHelper.cs:34-37`) — `Parallel.ForEach(rootChildren, MaxDegreeOfParallelism=4)`. UIA3/FlaUI COM-объекты не потокобезопасны для конкурентных вызовов по изменяющемуся дереву — редкие зависания/исключения, когда UI перерисовывается. Основной `click`-путь последовательный; задеты `ai-find`, `find-all`, `expand` (последний — как раз раскрытие «Подробности» в диалогах).
**Фикс:** `MaxDegreeOfParallelism=1` для этих путей или последовательный обход.

### 4. `--uia-only`-клик всё равно двигает физическую мышь
`MouseControl.ClickAtUia` (`MouseControl.cs:70-100`) вызывает FlaUI `element.Click()` — реальный клик с перемещением курсора, т.е. в headless/отключённой сессии бесполезен, несмотря на флаг `--uia-only`. Headless-безопасные пути в коде есть (`InvokePattern` в `InvokeCommand`, `BM_CLICK` в `Win32Helper.cs:44-53`), но не являются fallback-ом по умолчанию. `Drag` (`MouseControl.cs:118-133`) флаг `--uia-only` игнорирует полностью.
**Фикс:** клик-лестница по умолчанию `InvokePattern` → `BM_CLICK`/`PostMessage` → физическая мышь, с отчётом «какой способ сработал».

### 5. `SafetyGuard` защищает только `.rvs`-скрипты, не прямые команды
`SafetyGuard.IsDestructive`/`ConfirmDestructiveAction` вызываются только в `Commands/ScriptCommands.cs:208-210`. Диспетчер `Program.cs` не вызывает `SafetyGuard`; флаг `--non-interactive` (`Program.cs:322-323`) лишь ставит статик-свойство, но без проверки в диспетчере ни на что не влияет. Прямой `click "Purge Unused"` / `revit-api setParameter` идёт мимо защиты. Плюс сам детектор — `Contains` по 11 словам (`SafetyGuard.cs:11-16`): и ложные срабатывания, и пропуски.
**Фикс:** провести проверку через единый диспетчер команд, либо явно задокументировать, что защита работает только в скриптовом режиме.

### 6. Определение версии Revit по заголовку окна ломается на именах файлов с годом
`RevitVersionProfile.DetectVersion` (`RevitVersionProfile.cs:33-36`) и `RevitInstanceManager.DetectYearFromTitle` (`:198-204`) берут первое вхождение «2022…2027» в заголовке, где стоит и имя проекта. Файл `Объект_2025.rvt`, открытый в Revit 2023, определится как 2025.
**Фикс:** инвертировать приоритет — `FileVersionInfo` главного модуля процесса → реестр (`DetectFromRegistry` уже есть) → заголовок как последний вариант.

### 7. Автовыбор LLM-провайдера отправляет скриншот наружу без подтверждения
`LlmVisionClient.ResolveProvider` (`LlmVisionClient.cs:37-43,62-73`) без явного `--provider` берёт первого доступного по списку, а первый — RouterAI (`https://routerai.ru/...`). Достаточно оставленной в окружении переменной `ROUTERAI_API_KEY`, чтобы полный скриншот окна ушёл на сторонний сервис без подтверждения на вызов.
**Фикс:** сделать `--provider` обязательным явным выбором (не «первый доступный ключ») + предупреждение в лог при каждом сетевом вызове со скриншотом.

### 8. `revit-api`/`revit-select`/`revit-get` нефункциональны из коробки; пайп без аутентификации
Эти команды (`RevitApiCommand.cs`) работают через named pipe `\\.\pipe\ReVibe` к серверному аддину, которого в репозитории нет (`RevitApiCommand.cs:41` сам спрашивает «Is Revit running with the plugin loaded?»). Рукопожатие `PipeBridgeClient.cs:26-33` без аутентификации — статичный `loginUser="uictrl"`, ни токена, ни проверки отправителя: любой локальный процесс может слать команды записи параметров.
**Фикс:** в README пометить серверный аддин как обязательную отдельную зависимость; при живом сервере — добавить токен/проверку отправителя на пайп.

### 9. Мёртвый код и мелочи качества
- `RevitSession.cs` и `FlakyRetry.cs` — оба `[Obsolete]`, не удалены (~214 строк мёртвого кода).
- `PrintHelp` (`Program.cs:352-497`) дублирует README и уже разошёлся: `cv-match`/`cv-templates` перечислены дважды (`:466-480`).
- `LastOutput = Console.Out?.ToString()` (`Program.cs:271`) возвращает имя типа `TextWriter`, а не вывод команды.
- Нет `LICENSE`-файла — MIT заявлен только строкой в `README.md:824-826`; с полным текстом и именем правообладателя лицензирование чище.
- Автоскриншот base64 в stderr на каждый ненулевой exit-код (`Program.cs:286-291`) — сотни КБ в поток именно в момент ошибки; лучше писать PNG в файл и отдавать путь.

## Приоритет исправлений

1. Рабочий headless-скриншот (проблема 1) + watchdog на UIA (2) + убрать `Parallel.ForEach` (3) — базовая надёжность и headless-режим.
2. Клик-лестница `InvokePattern → PostMessage → мышь` (4) — чтобы `--uia-only` реально кликал без курсора.
3. Явный выбор LLM-провайдера (7) — приватность скриншотов.
4. `SafetyGuard` через диспетчер (5), версия по `FileVersionInfo` (6).
5. Гигиена: мёртвый код, справка, `LICENSE`, base64-в-stderr (9).

*Обзор подготовлен как внешний взгляд на код. Приоритет №1 (headless-скриншот) — для сценариев автоматизации Revit без активного экрана обычно ключевой.*
